## 1. 系統概述

### 1.1 系統目的

會議助理系統為一套內部使用之 AI 輔助工具，協助組織將會議音檔自動轉錄為逐字稿，並透過大型語言模型（LLM）萃取決議事項與待辦事項，降低人工整理會議記錄的時間成本。

### 1.2 系統範圍

| 項目 | 說明 |
|---|---|
| 使用對象 | 內部員工 |
| 預計使用人數 | 50 人 |
| 預計同時上線人數 | 20 人 |
| 部署方式 | 私有伺服器，不對外開放 |
| 資料存放 | 伺服器本地端（JSON 檔案） |

### 1.3 系統邊界

```
[使用者瀏覽器]
      │  HTTP (localhost 或內網)
      ▼
[FastAPI 後端 server.py]
      │              │
      ▼              ▼
[Whisper 轉錄]   [Ollama LLM 分析]
      │
      ▼
[storage.py → meetings/*.json]
```

系統目前不包含：使用者帳號管理、身份驗證、音檔長期儲存、Active Directory 整合（列為未來擴充）。

## 2. 系統架構

### 2.1 技術堆疊

| 層次 | 技術 | 版本 |
|---|---|---|
| 後端框架 | FastAPI | 0.111.0 |
| WSGI 伺服器 | Uvicorn | 0.29.0 |
| 語音轉文字 | OpenAI Whisper（本機執行） | base 模型，language="zh" |
| AI 分析 | Ollama + llama3.1:8b | 本機執行 |
| 資料儲存 | JSON 檔案 | — |
| 前端 | Vanilla HTML / CSS / JavaScript | — |
| 測試框架 | pytest + anyio[trio] | pytest 8.2.0 |
| Python 版本 | Python 3.11+ | — |

### 2.2 處理流程

```
使用者上傳音檔（或錄音）
        │
        ▼
格式與大小驗證（副檔名白名單、200 MB 上限）
        │
        ▼
非同步轉錄作業（背景執行緒）
  ├─ Whisper 語音轉文字（中文）→ 段落文字陣列
  └─ Ollama LLM 分析逐字稿
       ├─ 萃取決議事項（content, rationale, related_people）
       ├─ 萃取待辦事項（content, assignee, deadline, priority）
       └─ 比對歷史待辦事項（自動標記已完成）
        │
        ▼
儲存 meeting JSON（meetings/YYYY-MM-DD_HH-MM.json）
        │
        ▼
前端輪詢取得結果並顯示
```

## 3. 功能需求

### 3.1 音檔上傳與轉錄

| 編號 | 需求描述 |
|---|---|
| F-01 | 系統支援使用者上傳音檔，接受格式：mp3、wav、m4a、ogg、flac、webm |
| F-02 | 系統限制單次上傳檔案大小上限為 200 MB，超出時回傳錯誤提示 |
| F-03 | 系統提供瀏覽器內錄音功能，錄製完成後可直接送出轉錄 |
| F-04 | 音檔由 Whisper（中文模式）於伺服器端本機轉錄，轉錄以背景執行緒非同步進行 |
| F-05 | 前端以輪詢方式（每 2 秒）查詢轉錄狀態，完成後自動顯示結果 |

### 3.2 AI 分析

| 編號 | 需求描述 |
|---|---|
| F-06 | 轉錄完成後，系統自動呼叫 Ollama（llama3.1:8b）分析逐字稿 |
| F-07 | AI 萃取決議事項，包含：決議內容、依據、相關人員 |
| F-08 | AI 萃取待辦事項，包含：事項內容、負責人、截止日期、優先順序（high/medium/low） |
| F-09 | AI 比對當前所有未完成待辦事項，自動標記本次會議已完成之事項 |
| F-10 | 若 Ollama 服務不可用，系統仍完成轉錄並儲存，AI 分析欄位以空值呈現，前端顯示警示訊息 |

### 3.3 會議管理

| 編號 | 需求描述 |
|---|---|
| F-11 | 系統以時間戳記為 ID（格式：`YYYY-MM-DD_HH-MM`）儲存每筆會議記錄 |
| F-12 | 使用者可瀏覽歷史會議清單，清單顯示：會議 ID、建立時間、決議數、待辦事項數、未完成待辦數 |
| F-13 | 使用者可查看單筆會議完整內容（逐字稿、決議、待辦） |
| F-14 | 使用者可刪除指定會議記錄（永久刪除） |

### 3.4 決議事項管理

| 編號 | 需求描述 |
|---|---|
| F-15 | 使用者可新增決議事項至指定會議 |
| F-16 | 使用者可編輯決議事項之內容、依據、相關人員、狀態 |
| F-17 | 決議事項 ID 由系統自動指派（格式：`d-N`，從 1 起） |
| F-18 | 新增之決議事項狀態預設為 `confirmed`，使用者無法於新增時覆蓋此值 |

### 3.5 待辦事項管理

| 編號 | 需求描述 |
|---|---|
| F-19 | 使用者可新增待辦事項至指定會議 |
| F-20 | 使用者可編輯待辦事項之內容、負責人、截止日期、優先順序、狀態 |
| F-21 | 使用者可將待辦事項標記為完成（resolve） |
| F-22 | 使用者可跨會議查詢所有未完成待辦事項 |
| F-23 | 待辦事項 ID 由系統自動指派（格式：`a-N`，從 1 起） |
| F-24 | 新增之待辦事項狀態預設為 `pending`，使用者無法於新增時覆蓋此值 |

## 4. 非功能需求

### 4.1 效能

| 編號 | 需求描述 |
|---|---|
| NF-01 | 一小時音檔（約 50 MB，m4a 格式）之 Whisper 轉錄時間，於建議硬體規格下應在 5 分鐘以內完成 |
| NF-02 | Ollama 分析單次逐字稿回應時間應在 60 秒以內（建議硬體規格） |
| NF-03 | REST API 一般查詢（非轉錄）回應時間應在 500 ms 以內 |
| NF-04 | 系統支援 20 人同時使用前端功能（瀏覽、編輯），但轉錄作業為序列處理，不支援同時多個轉錄 |

### 4.2 可用性

| 編號 | 需求描述 |
|---|---|
| NF-05 | 系統採常駐服務方式部署，目標可用率 99%（維護視窗除外） |
| NF-06 | 服務異常時，前端應顯示明確錯誤訊息，不得出現空白頁或未處理例外 |

### 4.3 安全性

| 編號 | 需求描述 |
|---|---|
| NF-07 | 所有 meeting_id 輸入以正則表達式驗證（`^\d{4}-\d{2}-\d{2}_\d{2}-\d{2}$`），防止路徑穿越攻擊 |
| NF-08 | 決議及待辦事項更新操作僅允許修改白名單欄位，防止任意欄位注入 |
| NF-09 | 前端所有動態渲染內容須經 HTML 跳脫（escapeHtml），防止 XSS 攻擊 |
| NF-10 | 敏感會議內容僅存放於私有伺服器，不傳送至外部雲端服務 |

### 4.4 可維護性

| 編號 | 需求描述 |
|---|---|
| NF-11 | 後端採模組化設計（server / storage / analyzer 各司其職） |
| NF-12 | 測試覆蓋率：儲存層（storage.py）及 API 層（server.py）均有自動化測試 |

## 5. 資料設計

### 5.1 儲存結構

```
meetings/
├── 2026-03-28_14-30.json
├── 2026-03-29_09-00.json
└── ...
```

每個 JSON 檔案對應一筆會議記錄，以會議 ID 為檔名。

### 5.2 會議記錄 JSON Schema

```json
{
  "id": "YYYY-MM-DD_HH-MM",
  "created_at": "YYYY-MM-DDTHH:MM:SS",
  "transcript": "會議逐字稿全文",
  "decisions": [
    {
      "id": "d-1",
      "content": "決議內容",
      "rationale": "決議依據",
      "related_people": ["姓名1", "姓名2"],
      "status": "confirmed"
    }
  ],
  "action_items": [
    {
      "id": "a-1",
      "content": "待辦事項內容",
      "assignee": "負責人姓名",
      "deadline": "YYYY-MM-DD",
      "priority": "high | medium | low",
      "status": "pending | done"
    }
  ]
}
```

### 5.3 欄位說明

**decisions 欄位**

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | string | 系統自動指派，格式 `d-N` |
| content | string | 決議內容（必填） |
| rationale | string | 決議依據 |
| related_people | array[string] | 相關人員清單 |
| status | string | 固定為 `confirmed` |

**action_items 欄位**

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | string | 系統自動指派，格式 `a-N` |
| content | string | 事項內容（必填） |
| assignee | string | 負責人 |
| deadline | string | 截止日期（YYYY-MM-DD） |
| priority | string | `high` / `medium` / `low`（必填） |
| status | string | `pending`（預設）/ `done` |

## 6. API 規格

基礎 URL：`http://<server>:8000`

### 6.1 轉錄相關

#### POST `/transcribe`

上傳音檔，啟動非同步轉錄作業。

**Request：** `multipart/form-data`

| 欄位 | 型別 | 說明 |
|---|---|---|
| file | File | 音檔（mp3/wav/m4a/ogg/flac/webm，上限 200 MB） |

**Response 200：**
```json
{ "job_id": "uuid-string" }
```

**Error：**
- `400` 不支援的檔案格式
- `413` 檔案超過 200 MB

---

#### GET `/status/{job_id}`

查詢轉錄作業狀態。

**Response 200：**
```json
{ "status": "pending | processing | done | error", "error": null }
```

**Error：** `404` job 不存在

---

#### GET `/result/{job_id}`

取得轉錄結果（作業完成後才可取得）。

**Response 200：**
```json
{
  "segments": [{ "id": 0, "text": "段落文字", "tag": null }],
  "decisions": [...],
  "action_items": [...],
  "resolved_item_ids": ["a-1"],
  "meeting_id": "2026-03-29_14-30",
  "ollama_available": true
}
```

**Error：** `202` 尚未完成、`404` job 不存在

---

### 6.2 會議管理

#### GET `/meetings`

取得所有會議摘要清單（依時間降序）。

**Response 200：**
```json
{
  "meetings": [
    {
      "id": "2026-03-29_14-30",
      "created_at": "2026-03-29T14:30:00",
      "decision_count": 2,
      "action_item_count": 3,
      "pending_action_item_count": 2
    }
  ]
}
```

---

#### GET `/meetings/{meeting_id}`

取得單筆會議完整資料。

**Response 200：** 完整 meeting JSON（含逐字稿、決議、待辦）

**Error：** `400` meeting_id 格式錯誤、`404` 不存在

---

#### DELETE `/meetings/{meeting_id}`

刪除指定會議記錄。

**Response 200：**
```json
{ "status": "deleted" }
```

**Error：** `404` 不存在

---

### 6.3 決議事項

#### POST `/meetings/{meeting_id}/decisions`

新增決議至指定會議。

**Request body：**
```json
{
  "content": "決議內容（必填）",
  "rationale": "依據",
  "related_people": ["姓名"]
}
```

**Response 200：** 新增之 decision 物件（含自動指派 id）

**Error：** `404` 會議不存在、`422` content 未提供

---

#### PATCH `/meetings/{meeting_id}/decisions/{decision_id}`

更新指定決議（僅允許修改 content、rationale、related_people、status）。

**Request body：** 欲更新之欄位（部分更新）

**Response 200：** 更新後之 decision 物件

**Error：** `404` 決議不存在

---

### 6.4 待辦事項

#### POST `/meetings/{meeting_id}/action-items`

新增待辦事項至指定會議。

**Request body：**
```json
{
  "content": "事項內容（必填）",
  "assignee": "負責人",
  "deadline": "YYYY-MM-DD",
  "priority": "high（必填）"
}
```

**Response 200：** 新增之 action_item 物件

**Error：** `404` 會議不存在、`422` content 或 priority 未提供

---

#### PATCH `/meetings/{meeting_id}/action-items/{item_id}`

更新指定待辦事項（僅允許修改 content、assignee、deadline、priority、status）。

**Response 200：** 更新後之 action_item 物件

**Error：** `404` 待辦事項不存在

---

#### POST `/action-items/{meeting_id}/{item_id}/resolve`

將指定待辦事項標記為完成（status → done）。

**Response 200：**
```json
{ "status": "done" }
```

**Error：** `404` 待辦事項不存在

---

#### GET `/action-items/pending`

取得所有會議中所有未完成待辦事項。

**Response 200：**
```json
{
  "items": [
    {
      "id": "a-1",
      "content": "...",
      "assignee": "...",
      "deadline": "...",
      "priority": "high",
      "status": "pending",
      "meeting_id": "2026-03-29_14-30",
      "meeting_date": "2026-03-29"
    }
  ]
}
```

---

### 6.5 健康檢查

#### GET `/health`

**Response 200：**
```json
{ "status": "ok" }
```

## 7. 模組說明

### 7.1 server.py — API 層

| 責任 | 說明 |
|---|---|
| HTTP 路由 | 定義所有 REST API 端點 |
| 輸入驗證 | 檔案格式、大小、必填欄位驗證 |
| 非同步轉錄 | 啟動背景執行緒執行 Whisper + Ollama |
| 作業狀態管理 | 以記憶體內 dict 管理轉錄作業狀態 |
| 跨域設定 | CORS（限內網環境） |

> **注意：** 作業狀態僅存於記憶體，伺服器重啟後作業狀態將清空。進行中的轉錄若遇服務重啟，需重新上傳音檔。

### 7.2 storage.py — 儲存層

| 責任 | 說明 |
|---|---|
| meeting CRUD | save / load / list / delete |
| 待辦事項操作 | get_pending / resolve / update |
| 決議事項操作 | add / update |
| 輸入驗證 | meeting_id 格式驗證（路徑穿越防護）、欄位更新白名單 |

### 7.3 analyzer.py — AI 分析層

| 責任 | 說明 |
|---|---|
| Prompt 建構 | 組合逐字稿與歷史待辦事項為 LLM 提示詞 |
| Ollama 呼叫 | HTTP 呼叫本機 Ollama（最多 3 次重試） |
| 回應解析 | 解析 LLM JSON 輸出，支援 code block 格式 |
| 降級處理 | Ollama 不可用時回傳空結果，不影響轉錄儲存 |

### 7.4 index.html — 上傳頁面

| 功能 | 說明 |
|---|---|
| 音檔上傳 | 選擇本機檔案，含格式與大小前端驗證 |
| 瀏覽器錄音 | 使用 MediaRecorder API 錄製 webm 音檔 |
| 結果顯示 | 顯示逐字稿段落、決議事項、待辦事項 |
| 輪詢機制 | 每 2 秒查詢作業狀態，完成後停止輪詢 |

### 7.5 history.html — 歷史記錄頁面

| 功能 | 說明 |
|---|---|
| 會議清單 | 列出所有歷史會議，顯示摘要資訊 |
| 會議詳情 | 點選後顯示逐字稿、決議、待辦事項 |
| 決議管理 | 新增、編輯決議事項（行內編輯） |
| 待辦管理 | 新增、編輯、完成待辦事項 |
| 跨會議待辦 | 顯示所有未完成待辦事項（含來源會議） |

## 8. 硬體建議規格

### 8.1 最低規格（可運作，效能有限）

| 項目 | 規格 |
|---|---|
| CPU | Intel/AMD 8 核心 |
| RAM | 32 GB |
| GPU | NVIDIA GPU，VRAM ≥ 8 GB（如 RTX 3060） |

| 項目 | 規格 | 說明 |
|---|---|---|
| CPU | Intel/AMD 16 核心 | 提升並發回應能力 |

| GPU | NVIDIA GPU，VRAM ≥ 16 GB（如 RTX 4080、A4000） | 可同時加速 Whisper 與 Ollama |
| 儲存 | SSD 500 GB | 預留音檔暫存與會議資料成長空間 |
| 作業系統 | Ubuntu 22.04 LTS | |
| CUDA | 12.x | 配合 PyTorch GPU 支援 |

### 8.3 模型儲存空間需求

| 模型 | 大小 |
|---|---|
| Whisper base | ~145 MB |
| Whisper large-v3（選用） | ~1.5 GB |
| llama3.1:8b（Q4_0） | ~4.7 GB |

## 9. 部署步驟

### 9.1 前置作業

1. 安裝 Ubuntu 22.04 LTS，設定靜態 IP
2. 安裝 NVIDIA Driver + CUDA 12.x（若有 GPU）
3. 安裝 Python 3.11+

```bash
sudo apt update && sudo apt install -y python3.11 python3.11-venv python3-pip ffmpeg
```

4. 安裝 Ollama

```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama pull llama3.1:8b
```

### 9.2 應用程式部署

```bash
# 1. 複製專案至伺服器
git clone <repo-url> /opt/meeting-assistant
cd /opt/meeting-assistant

# 2. 建立虛擬環境並安裝相依套件
python3.11 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 3. 建立資料目錄
mkdir -p meetings

# 4. 啟動服務
uvicorn server:app --host 0.0.0.0 --port 8000
```

### 9.3 常駐服務設定（systemd）

建立 `/etc/systemd/system/meeting-assistant.service`：

```ini
[Unit]
Description=Meeting Assistant API
After=network.target

[Service]
Type=simple
User=<service-user>
WorkingDirectory=/opt/meeting-assistant
ExecStart=/opt/meeting-assistant/venv/bin/uvicorn server:app --host 0.0.0.0 --port 8000
Restart=on-failure
RestartSec=5
```

## 10. 安全設計

### 10.1 輸入驗證

| 防護目標 | 實作方式 |
|---|---|
| 路徑穿越攻擊 | meeting_id 以正則 `^\d{4}-\d{2}-\d{2}_\d{2}-\d{2}$` 驗證，不符合格式的請求直接拒絕 |
| 任意欄位注入 | update_decision / update_action_item 僅處理白名單欄位（content, rationale, related_people, status / content, assignee, deadline, priority, status） |
| 前端 XSS | 所有動態渲染內容透過 escapeHtml() 跳脫（`&`, `<`, `>`, `"`, `'`） |
| 檔案上傳 | 副檔名白名單（mp3/wav/m4a/ogg/flac/webm）+ 大小上限 200 MB |

### 10.2 資料保護

- 所有會議資料（含逐字稿、決議、待辦）儲存於私有伺服器本機，不傳送至外部服務
- Whisper 與 Ollama 均於本機執行，音檔及文字內容不離開伺服器
- 建議伺服器存取限制於內部網路，不對外部 Internet 開放

### 10.3 操作稽核（未來擴充，現階段規劃）

系統目前無身份驗證機制，後續擴充時需加入：

| 功能 | 說明 |
|---|---|
| 使用者登入 / 登出 | 接軌 Active Directory（LDAP/OAuth） |
| 操作日誌 | 記錄使用者登入、登出、新增/刪除/修改操作，含時間戳與 IP |
| 日誌格式 | 結構化日誌（JSON），便於後續查詢 |
| 日誌保存 | 建議保存 1 年以上 |
