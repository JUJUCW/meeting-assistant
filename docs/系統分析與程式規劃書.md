## 1. 系統概述

### 1.1 系統目的

會議助理系統為一套內部使用之 AI 輔助工具，協助組織將會議音檔自動轉錄為逐字稿，並透過大型語言模型（LLM）萃取決議事項、待辦事項及會議摘要，降低人工整理會議記錄的時間成本。

### 1.2 系統範圍

| 項目 | 說明 |
|---|---|
| 使用對象 | 內部員工 |
| 預計使用人數 | 50 人 |
| 預計同時上線人數 | 20 人 |
| 部署方式 | 私有伺服器，不對外開放 |
| 資料存放 | 伺服器本地端（JSON 檔案） |

### 1.3 系統邊界

```
[使用者瀏覽器 — Vue 3 SPA]
      │  HTTP (localhost 或內網)
      ▼
[FastAPI 後端 server.py]
      │              │
      ▼              ▼
[faster-whisper]  [Ollama LLM 分析 / 摘要]
      │
      ▼
[storage.py → meetings/*.json]
```

系統目前不包含：使用者帳號管理、身份驗證、音檔長期儲存、Active Directory 整合（列為未來擴充）。

## 2. 系統架構

### 2.1 技術堆疊

| 層次 | 技術 | 版本 / 說明 |
|---|---|---|
| 後端框架 | FastAPI | 0.111.0 |
| WSGI 伺服器 | Uvicorn | 0.29.0 |
| 語音轉文字 | faster-whisper（本機執行） | turbo 模型，int8 量化，language="zh" |
| 繁簡轉換 | OpenCC | s2t（簡轉繁） |
| AI 分析 / 摘要 | Ollama + llama3.1:8b | 本機執行 |
| 文件匯出 | python-docx | .docx 格式產生 |
| 資料儲存 | JSON 檔案 | meetings/*.json |
| 前端框架 | Vue 3 + Vite + TypeScript | SPA，Vue Router |
| 測試框架 | pytest + anyio[trio] | pytest 8.2.0 |
| Python 版本 | Python 3.11+ | — |

### 2.2 處理流程

```
使用者上傳音檔（或錄音）
        │
        ▼
格式與大小驗證（副檔名白名單、200 MB 上限）
        │
        ▼
立即回傳 job_id，前端導向歷史記錄頁面
        │
        ▼
非同步轉錄作業（背景執行緒）
  ├─ faster-whisper 語音轉文字（turbo、int8、中文）→ 段落文字陣列
  ├─ OpenCC 簡轉繁
  └─ Ollama LLM 分析逐字稿
       ├─ 萃取決議事項（content, rationale, related_people）
       ├─ 萃取待辦事項（content, assignee, deadline, priority）
       └─ 比對歷史待辦事項（自動標記已完成）
        │
        ▼
儲存 meeting JSON（meetings/YYYY-MM-DD_HH-MM.json）
        │
        ▼
前端顯示進度橫幅（輪詢 /status/{job_id} 每 2 秒）
轉錄完成後自動刷新會議清單

[使用者進入會議詳情後，可按需產生摘要]
        │
        ▼
POST /meetings/{id}/summary → Ollama 產生摘要 → 存入 JSON
```

## 3. 功能需求

### 3.1 音檔上傳與轉錄

| 編號 | 需求描述 |
|---|---|
| F-01 | 系統支援使用者上傳音檔，接受格式：mp3、wav、m4a、ogg、flac、webm |
| F-02 | 系統限制單次上傳檔案大小上限為 200 MB，超出時回傳錯誤提示 |
| F-03 | 系統提供瀏覽器內錄音功能，錄製完成後可直接送出轉錄 |
| F-04 | 音檔由 faster-whisper（turbo 模型，int8 量化，中文模式）於伺服器端本機轉錄，轉錄以背景執行緒非同步進行；模型於服務啟動時一次性載入 |
| F-05 | 送出音檔後前端立即導向歷史記錄頁，轉錄狀態以進度橫幅顯示（含已等待時間與預計剩餘時間）；前端每 2 秒輪詢作業狀態，完成後自動刷新清單 |

### 3.2 AI 分析

| 編號 | 需求描述 |
|---|---|
| F-06 | 轉錄完成後，系統自動呼叫 Ollama（llama3.1:8b）分析逐字稿 |
| F-07 | AI 萃取決議事項，包含：決議內容、依據、相關人員 |
| F-08 | AI 萃取待辦事項，包含：事項內容、負責人、截止日期、優先順序（high/medium/low） |
| F-09 | AI 比對當前所有未完成待辦事項，自動標記本次會議已完成之事項 |
| F-10 | 若 Ollama 服務不可用，系統仍完成轉錄並儲存，AI 分析欄位以空值呈現 |

### 3.3 會議摘要

| 編號 | 需求描述 |
|---|---|
| F-11 | 使用者可於會議詳情頁按需產生 AI 摘要（POST /meetings/{id}/summary） |
| F-12 | 摘要由 Ollama 依逐字稿產生，以 3–5 句繁體中文呈現 |
| F-13 | 已有摘要的會議可重新產生（覆蓋舊摘要） |
| F-14 | 若 Ollama 不可用，產生摘要操作回傳 503 錯誤，前端顯示提示 |

### 3.4 會議管理

| 編號 | 需求描述 |
|---|---|
| F-15 | 系統以時間戳記為 ID（格式：`YYYY-MM-DD_HH-MM`）儲存每筆會議記錄 |
| F-16 | 使用者可瀏覽歷史會議清單，清單顯示：標題、建立時間、決議數、待辦事項數、未完成待辦數、標籤、分類 |
| F-17 | 使用者可查看單筆會議完整內容（逐字稿、摘要、決議、待辦） |
| F-18 | 使用者可刪除指定會議記錄（永久刪除） |
| F-19 | 使用者可編輯會議標題（行內編輯） |
| F-20 | 使用者可對會議設定分類（單選）及標籤（多選） |
| F-21 | 使用者可依分類或標籤篩選會議清單 |
| F-22 | 使用者可以關鍵字搜尋會議（逐字稿、決議、待辦事項） |

### 3.5 決議事項管理

| 編號 | 需求描述 |
|---|---|
| F-23 | 使用者可新增決議事項至指定會議 |
| F-24 | 使用者可編輯決議事項之內容、依據、相關人員、狀態 |
| F-25 | 決議事項 ID 由系統自動指派（格式：`d-N`，從 1 起） |
| F-26 | 新增之決議事項狀態預設為 `confirmed` |

### 3.6 待辦事項管理

| 編號 | 需求描述 |
|---|---|
| F-27 | 使用者可新增待辦事項至指定會議 |
| F-28 | 使用者可編輯待辦事項之內容、負責人、截止日期、優先順序、狀態 |
| F-29 | 使用者可將待辦事項標記為完成 |
| F-30 | 使用者可跨會議查詢所有未完成待辦事項 |
| F-31 | 待辦事項 ID 由系統自動指派（格式：`a-N`，從 1 起） |

### 3.7 匯出

| 編號 | 需求描述 |
|---|---|
| F-32 | 使用者可將單筆會議匯出為 Markdown 格式（.md） |
| F-33 | 使用者可將單筆會議匯出為 JSON 格式（.json） |
| F-34 | 使用者可將單筆會議匯出為 Word 文件格式（.docx），內容包含：標題、日期、標籤、摘要、決策、待辦事項 |

## 4. 非功能需求

### 4.1 效能

| 編號 | 需求描述 |
|---|---|
| NF-01 | 一小時音檔（約 50 MB，m4a 格式）之轉錄時間，於建議硬體規格下應在 5 分鐘以內完成（faster-whisper turbo + int8） |
| NF-02 | Ollama 分析單次逐字稿回應時間應在 60 秒以內（建議硬體規格） |
| NF-03 | REST API 一般查詢（非轉錄）回應時間應在 500 ms 以內 |
| NF-04 | 系統支援 20 人同時使用前端功能（瀏覽、編輯），但轉錄作業為序列處理，不支援同時多個轉錄 |

### 4.2 可用性

| 編號 | 需求描述 |
|---|---|
| NF-05 | 系統採常駐服務方式部署，目標可用率 99%（維護視窗除外） |
| NF-06 | 服務異常時，前端應顯示明確錯誤訊息，不得出現空白頁或未處理例外 |

### 4.3 安全性

| 編號 | 需求描述 |
|---|---|
| NF-07 | 所有 meeting_id 輸入以正則表達式驗證（`^\d{4}-\d{2}-\d{2}_\d{2}-\d{2}$`），防止路徑穿越攻擊 |
| NF-08 | 決議及待辦事項更新操作僅允許修改白名單欄位，防止任意欄位注入 |
| NF-09 | 前端所有動態渲染內容由 Vue 自動跳脫，防止 XSS 攻擊 |
| NF-10 | 敏感會議內容僅存放於私有伺服器，不傳送至外部雲端服務 |

### 4.4 可維護性

| 編號 | 需求描述 |
|---|---|
| NF-11 | 後端採模組化設計（server / storage / analyzer 各司其職） |
| NF-12 | 前端採元件化設計（Vue 3 SPA，composables 抽離業務邏輯） |
| NF-13 | 測試覆蓋率：儲存層（storage.py）及 API 層（server.py）均有自動化測試 |

## 5. 資料設計

### 5.1 儲存結構

```
meetings/
├── 2026-03-28_14-30.json
├── 2026-03-29_09-00.json
└── ...
categories.json
```

### 5.2 會議記錄 JSON Schema

```json
{
  "id": "YYYY-MM-DD_HH-MM",
  "created_at": "YYYY-MM-DDTHH:MM:SS",
  "title": "會議標題（選填）",
  "transcript": "會議逐字稿全文",
  "summary": "AI 產生之會議摘要（選填）",
  "category_id": "cat-1（選填）",
  "tags": ["標籤A", "標籤B"],
  "decisions": [
    {
      "id": "d-1",
      "content": "決議內容",
      "rationale": "決議依據",
      "related_people": ["姓名1", "姓名2"],
      "status": "confirmed"
    }
  ],
  "action_items": [
    {
      "id": "a-1",
      "content": "待辦事項內容",
      "assignee": "負責人姓名",
      "deadline": "YYYY-MM-DD",
      "priority": "high | medium | low",
      "status": "pending | done"
    }
  ]
}
```

### 5.3 分類 JSON Schema（categories.json）

```json
[
  { "id": "cat-1", "name": "週會" },
  { "id": "cat-2", "name": "產品" }
]
```

### 5.4 欄位說明

**會議頂層欄位**

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | string | 時間戳記格式 `YYYY-MM-DD_HH-MM` |
| created_at | string | ISO 8601 時間戳記 |
| title | string | 使用者自訂標題（選填） |
| transcript | string | Whisper 轉錄全文 |
| summary | string | AI 產生摘要（選填，按需產生） |
| category_id | string | 對應 categories.json 的分類 ID（選填） |
| tags | array[string] | 使用者自訂標籤清單 |

**decisions 欄位**

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | string | 系統自動指派，格式 `d-N` |
| content | string | 決議內容（必填） |
| rationale | string | 決議依據 |
| related_people | array[string] | 相關人員清單 |
| status | string | 固定為 `confirmed` |

**action_items 欄位**

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | string | 系統自動指派，格式 `a-N` |
| content | string | 事項內容（必填） |
| assignee | string | 負責人 |
| deadline | string | 截止日期（YYYY-MM-DD） |
| priority | string | `high` / `medium` / `low`（必填） |
| status | string | `pending`（預設）/ `done` |

## 6. API 規格

基礎 URL：`http://<server>:8000`

### 6.1 轉錄相關

#### POST `/transcribe`

上傳音檔，啟動非同步轉錄作業。

**Request：** `multipart/form-data`

| 欄位 | 型別 | 說明 |
|---|---|---|
| file | File | 音檔（mp3/wav/m4a/ogg/flac/webm，上限 200 MB） |

**Response 200：**
```json
{ "job_id": "uuid-string" }
```

**Error：** `400` 不支援的檔案格式、`413` 檔案超過 200 MB

---

#### GET `/status/{job_id}`

查詢轉錄作業狀態。

**Response 200：**
```json
{ "status": "pending | processing | done | error", "error": null }
```

**Error：** `404` job 不存在

---

#### GET `/result/{job_id}`

取得轉錄結果（作業完成後才可取得）。

**Response 200：**
```json
{
  "segments": [{ "id": 0, "text": "段落文字", "tag": null }],
  "decisions": [...],
  "action_items": [...],
  "resolved_item_ids": ["a-1"],
  "meeting_id": "2026-03-29_14-30",
  "ollama_available": true
}
```

**Error：** `202` 尚未完成、`404` job 不存在

---

### 6.2 會議管理

#### GET `/meetings`

取得所有會議摘要清單（依時間降序）。

**Response 200：**
```json
{
  "meetings": [
    {
      "id": "2026-03-29_14-30",
      "created_at": "2026-03-29T14:30:00",
      "title": "週會",
      "tags": ["重要"],
      "category_id": "cat-1",
      "decision_count": 2,
      "action_item_count": 3,
      "pending_action_item_count": 2
    }
  ]
}
```

---

#### GET `/meetings/search?q={keyword}`

依關鍵字搜尋會議（逐字稿、決議、待辦事項）。

**Response 200：** 同 `/meetings`，每筆附加 `hits` 陣列（命中欄位與摘要片段）

---

#### GET `/meetings/{meeting_id}`

取得單筆會議完整資料。

**Response 200：** 完整 meeting JSON

**Error：** `400` meeting_id 格式錯誤、`404` 不存在

---

#### DELETE `/meetings/{meeting_id}`

刪除指定會議記錄。

**Response 200：** `{ "status": "deleted" }`

**Error：** `404` 不存在

---

#### PATCH `/meetings/{meeting_id}/title`

更新會議標題。

**Request body：** `{ "title": "新標題" }`

**Response 200：** 更新後的完整 meeting JSON

---

#### PATCH `/meetings/{meeting_id}/tags`

更新會議分類與標籤。

**Request body：** `{ "category_id": "cat-1", "tags": ["標籤A"] }`（部分更新）

**Response 200：** 更新後的完整 meeting JSON

---

### 6.3 會議摘要與匯出

#### POST `/meetings/{meeting_id}/summary`

呼叫 Ollama 產生（或重新產生）會議摘要，儲存後回傳。

**Response 200：** `{ "summary": "摘要文字" }`

**Error：** `404` 會議不存在、`422` 無逐字稿、`503` Ollama 不可用

---

#### GET `/meetings/{meeting_id}/export/docx`

產生並下載 Word 文件（.docx）。內容包含：標題、日期、標籤、摘要、決策、待辦事項。

**Response 200：** `application/vnd.openxmlformats-officedocument.wordprocessingml.document`

**Error：** `404` 會議不存在

---

### 6.4 決議事項

#### POST `/meetings/{meeting_id}/decisions`

新增決議至指定會議。

**Request body：**
```json
{
  "content": "決議內容（必填）",
  "rationale": "依據",
  "related_people": ["姓名"]
}
```

**Response 200：** 新增之 decision 物件

**Error：** `404` 會議不存在、`422` content 未提供

---

#### PATCH `/meetings/{meeting_id}/decisions/{decision_id}`

更新指定決議（僅允許修改 content、rationale、related_people、status）。

**Response 200：** 更新後之 decision 物件

**Error：** `404` 決議不存在

---

### 6.5 待辦事項

#### POST `/meetings/{meeting_id}/action-items`

新增待辦事項至指定會議。

**Request body：**
```json
{
  "content": "事項內容（必填）",
  "assignee": "負責人",
  "deadline": "YYYY-MM-DD",
  "priority": "high（必填）"
}
```

**Response 200：** 新增之 action_item 物件

**Error：** `404` 會議不存在、`422` content 或 priority 未提供

---

#### PATCH `/meetings/{meeting_id}/action-items/{item_id}`

更新指定待辦事項（僅允許修改 content、assignee、deadline、priority、status）。

**Response 200：** 更新後之 action_item 物件

**Error：** `404` 待辦事項不存在

---

#### POST `/action-items/{meeting_id}/{item_id}/resolve`

將指定待辦事項標記為完成（status → done）。

**Response 200：** `{ "status": "done" }`

---

#### GET `/action-items/pending`

取得所有會議中所有未完成待辦事項。

**Response 200：**
```json
{
  "items": [
    {
      "id": "a-1",
      "content": "...",
      "assignee": "...",
      "deadline": "...",
      "priority": "high",
      "status": "pending",
      "meeting_id": "2026-03-29_14-30",
      "meeting_date": "2026-03-29"
    }
  ]
}
```

---

### 6.6 分類管理

#### GET `/categories`

取得所有分類清單。

**Response 200：** `[{ "id": "cat-1", "name": "週會" }, ...]`

---

#### POST `/categories`

新增分類。

**Request body：** `{ "name": "分類名稱" }`

**Response 200：** 新增之分類物件（含自動指派 id）

## 9. 部署步驟

### 9.1 前置作業

1. 安裝 Ubuntu 22.04 LTS，設定靜態 IP
2. 安裝 NVIDIA Driver + CUDA 12.x（若有 GPU）
3. 安裝 Python 3.11+、Node.js 18+、ffmpeg

```bash
sudo apt update && sudo apt install -y python3.11 python3.11-venv python3-pip nodejs npm ffmpeg
```

4. 安裝 Ollama

```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama pull llama3.1:8b
```

### 9.2 應用程式部署

```bash
# 1. 複製專案至伺服器
git clone <repo-url> /opt/meeting-assistant
cd /opt/meeting-assistant

# 2. 建立虛擬環境並安裝後端相依套件
python3.11 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 3. 建置前端
cd frontend && npm install && npm run build
cd ..

# 4. 建立資料目錄
mkdir -p meetings

# 5. 啟動服務（前端 build 產物由 FastAPI 或 nginx 提供）
uvicorn server:app --host 0.0.0.0 --port 8000
```

### 9.3 常駐服務設定（systemd）

建立 `/etc/systemd/system/meeting-assistant.service`：

```ini
[Unit]
Description=Meeting Assistant API
After=network.target

[Service]
Type=simple
User=<service-user>
WorkingDirectory=/opt/meeting-assistant
ExecStart=/opt/meeting-assistant/venv/bin/uvicorn server:app --host 0.0.0.0 --port 8000
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable meeting-assistant
sudo systemctl start meeting-assistant
```

### 9.4 確認部署正常

```bash
# API 健康檢查
curl http://localhost:8000/health
# 預期：{"status": "ok"}

# 確認 Ollama
curl http://localhost:11434/api/tags
# 預期：包含 llama3.1:8b 模型資訊
```

## 10. 安全設計

### 10.1 輸入驗證

| 防護目標 | 實作方式 |
|---|---|
| 路徑穿越攻擊 | meeting_id 以正則 `^\d{4}-\d{2}-\d{2}_\d{2}-\d{2}$` 驗證，不符合格式的請求直接拒絕 |
| 任意欄位注入 | update_decision / update_action_item / update_meeting_tags 僅處理白名單欄位 |
| 前端 XSS | Vue 3 預設對模板綁定進行 HTML 跳脫 |
| 檔案上傳 | 副檔名白名單（mp3/wav/m4a/ogg/flac/webm）+ 大小上限 200 MB |

### 10.2 資料保護

- 所有會議資料（含逐字稿、決議、待辦、摘要）儲存於私有伺服器本機，不傳送至外部服務
- faster-whisper 與 Ollama 均於本機執行，音檔及文字內容不離開伺服器
- 建議伺服器存取限制於內部網路，不對外部 Internet 開放

### 10.3 操作稽核（未來擴充，現階段規劃）

| 功能 | 說明 |
|---|---|
| 使用者登入 / 登出 | 接軌 Active Directory（LDAP/OAuth） |
| 操作日誌 | 記錄使用者登入、登出、新增/刪除/修改操作，含時間戳與 IP |
| 日誌格式 | 結構化日誌（JSON） |
| 日誌保存 | 建議保存 1 年以上 |
